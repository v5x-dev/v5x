Plan 02 — CLI reliability and UX

Goal: align the Web Serial adapter with runtime reality and reject unusable inputs before expensive work.

1. Correct stream lifecycle modeling

File: packages/cli/src/adapter.ts

  1. Make readable and writable streams nullable while a port is closed.
  2. Remove casts that turn null into non-null stream types.
  3. Update the shared serial interfaces or use the platform Web Serial declarations directly where possible.
  4. Move stream nulling and listener removal into finally so native close errors cannot leave an apparently open adapter.
  5. Define and test how native data/error events behave after cancellation.

Acceptance: TypeScript and runtime agree about closed ports; close failures still leave reusable internal state.

2. Validate upload artifacts early

Files: utils/upload.ts, utils/project.ts

  1. Stat artifacts before reading them.
  2. Reject empty files and files above the applicable protocol/target size limit.
  3. Validate both hot and cold PROS packages.
  4. Report the offending path, actual size, and supported limit.
  5. Only connect to hardware after all project inspection and local validation succeeds.

Acceptance: invalid artifacts fail without opening a serial port or loading the entire file.

3. Generate valid projects

Files: utils/scaffold.ts, commands/new.ts, commands/init.ts

  1. Separate destination path, display name, Cargo package name, and PROS remote name.
  2. Validate names using toolchain-specific rules and return actionable errors.
  3. Reject path separators when the argument is documented as a name; use an explicit path option for nested destinations.
  4. Keep staging, reservation, and concurrent-write protections intact.
  5. Add fixture validation for generated Cargo TOML and PROS metadata.

Acceptance: every accepted scaffold can be parsed and inspected as its requested project type.

4. Normalize command output

Files: commands/devices.ts, commands/dir.ts, upload progress code

  1. Reuse the serial package’s firmware version model instead of duplicating bit decoding.
  2. Choose and document UTC or local timestamps; format using an explicit locale and timezone policy.
  3. Handle unknown smart-device values with a stable fallback label.
  4. Keep machine-readable data on stdout and progress/diagnostics on stderr.
  5. Disable terminal control sequences when output is not a TTY where applicable.

Acceptance: output is deterministic enough for snapshots and piping, with no library-owned logging mixed into it.

Required tests

  • Closed, open, errored, and reopened adapter stream states.
  • Native close rejection still clears adapter state.
  • Empty and oversized hot/cold artifacts.
  • Invalid Cargo/PROS names and nested-path attempts.
  • Unknown device types and deterministic date/version formatting.
