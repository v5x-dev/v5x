Plan 01 — Serial correctness and lifecycle

Goal: make transfers atomic, async behavior observable, refreshes consistent, and connection cleanup deterministic.

1. Make protocol transfers atomic

Files: VexConnection.ts, VexDeviceState.ts, VexTransfers.ts

  1. Expose one private connection-level transaction primitive that queues an entire operation.
  2. Split upload/download implementations into locked public entry points and explicitly named unlocked helpers.
  3. Hold one transaction across INI, cold, and binary program uploads.
  4. Route erase, clear, screenshot setup/download, and other file-transfer-mode operations through the same queue.
  5. Keep device-state transfer depth only for refresh pausing; rename it so it is not mistaken for a mutex.
  6. Ensure every entered transfer mode exits from finally, preserving the original error when cleanup also fails.

Acceptance: concurrent program upload, read, erase, and capture operations execute in request order with no packet interleaving.

2. Replace fire-and-forget setters

Files: VexDevice.ts, VexDeviceState.ts, documentation pages referencing program or match state

  1. Add explicit methods such as runProgram, stopProgram, and setMatchMode returning promises.
  2. Deprecate property setters for one release if compatibility is required; otherwise remove them in the next breaking version.
  3. Update state only after the device acknowledges the command.
  4. Document rejection behavior and disconnected-device behavior.

Acceptance: callers can await success and catch command failures; no rejected promise is detached.

3. Harden connection lifecycle

Files: VexConnection.ts, CLI adapter.ts

  1. Store the port disconnect callback and remove it during close.
  2. Represent in-progress close with a shared promise so concurrent callers await the same cleanup.
  3. Clear reader, writer, port, callbacks, listeners, and flags in deterministic finally blocks.
  4. Emit one disconnected event per connected lifecycle.
  5. Remove the one-millisecond lock-release delay if stream release ordering can be expressed directly; otherwise isolate and document it.

Acceptance: repeated open/close cycles do not grow listener counts and concurrent closes resolve only after cleanup completes.

4. Make polling and refresh state safe

Files: VexFirmware.ts, VexDevice.ts

  1. Replace interval-based sleepUntil with a loop that catches predicate errors, clears timers, and rejects.
  2. Validate non-negative timeout and positive interval values.
  3. Build a complete refresh snapshot before mutating public state.
  4. Commit only after all required replies are received; on failure mark availability accurately without mixing refresh generations.
  5. Prevent refresh work from continuing after disposal.

Acceptance: throwing predicates reject promptly and failed mid-refresh requests cannot expose partially updated state.

5. Bound firmware and screen operations

Files: VexFirmware.ts, VexTransfers.ts

  1. Stream firmware downloads while enforcing compressed byte limits and validating declared content length.
  2. Set per-entry and aggregate extracted limits before reading ZIP entry buffers.
  3. Reject encrypted, empty, unexpected, or oversized images before upload.
  4. Validate the screen-capture command reply before downloading the framebuffer.
  5. Move controller channel status output from console.log to progress callbacks or events.

Acceptance: oversized firmware fails before allocation exceeds configured limits; a rejected screen request performs no download.

Required regression tests

  • Whole-program upload blocks concurrent transfers until all related files finish.
  • Transfer cleanup runs after every failure point.
  • Predicate exceptions reject without leaked timers.
  • Repeated reconnects retain exactly one active disconnect listener.
  • Concurrent close calls await one cleanup.
  • Partial refresh failure preserves the previous coherent snapshot.
  • Firmware compressed and extracted size limits are enforced.
  • Screen capture rejects NACK and timeout replies.
